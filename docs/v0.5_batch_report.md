### Surface-quality fix (public-title cleaning)

A reviewer pass found that scraped SH list-page link/event-handler residue (HTML/JS markup that the SH
portal embeds in its visible-text titles) had leaked into **89** v0.5 question *display titles*. The
grounding verifier did not catch it because grounding only checks evidence, not question surface text.
Fix applied:

- `scripts/ingest_providers_v05.py` now runs a `clean_title()` (unescape HTML entities, strip the
  list-page handler markup + tags/attributes, normalize whitespace) on every scraped title upstream;
  `scripts/build_qa_providers_v05.py` repeats the sanitize defensively. Answer content is unchanged —
  only the human-readable announcement title in the question was cleaned.
- `workspace_local/processed/sh-announcements/**` and `workspace_local/audit/index_providers_v05.json`
  were regenerated **memory-safely, one file per isolated process** (`--row`/`--merge-index`; an
  earlier single-process re-ingest hit a 120s timeout / memory pressure). `data/qa_v0.5_candidates.jsonl`
  was regenerated; counts and answers are identical, titles are now clean.
- New public-surface gate in `scripts/validate_dataset.py` (`assert_no_surface_artifacts`) scans
  `data/`, `docs/`, `README.md`, `DATASET_CARD.md` for these scraped-markup tokens and fails the build
  if any are found.

Verification: `rg <markup-tokens> data docs README.md DATASET_CARD.md` → **0 matches**; validate/verify
exit 0; readiness `--allow-dev` exit 0 (still `dev/seed only`). v0.5 remains **not public-ready**.
