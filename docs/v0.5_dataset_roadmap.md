# v0.5 Dataset Roadmap

Status date: 2026-06-05

Superseded for post-`v0.6.3` planning: use `docs/v1_roadmap.md` as the current roadmap. This document is
retained as historical planning context for the v0.5/v0.6 source-expansion work.

This roadmap prioritizes dataset quality before paper writing. The goal is not to claim a
"perfect" benchmark in an absolute sense, but to turn v0.4-public from a strong LH-centered public
seed into a Korean real-world housing benchmark that is harder to dismiss on coverage, validity,
or release-safety grounds.

## 1. Current Baseline

v0.4-public is already useful:

- 812 verified QA.
- 10 official LH announcements.
- 3 housing statutes.
- MOLIT apartment trade rows and HUG sale-history rows.
- 32k/64k/128k/256k/512k context tiers.
- Automated gates pass: schema validation, QA verification, public-readiness, leakage scan.

The remaining weaknesses are not mainly answer correctness. They are benchmark validity gaps:

- Single announcement provider: LH only.
- Limited regional coverage: 3 시도, mostly 경기.
- All QA currently use `dev` split.
- No hidden/test split for leaderboard-style evaluation.
- Some task families are under target from the v0.4 plan.
- Context tiers are imbalanced, especially 64k.
- Public docs still contain some stale v0.3/seed language.
- Human review sample is drawn but not signed off.

## 2. Target Framing

Use this framing for v0.5 and later:

> Korean real-world housing long-context benchmark for evaluating full-context LLMs, RAG systems,
> and table/tool pipelines on official housing announcements, public tabular data, and housing law.

Do not frame it as:

- a general Korean long-context benchmark;
- a replacement for LongBench or InfiniteBench;
- hallucination-free;
- complete coverage of Korean housing policy;
- legal or housing-eligibility advice.

## 3. Non-Negotiable Release Policy

Public release may include:

- QA items.
- Short answers.
- Source URLs.
- Provider/region/year/type metadata.
- Page, row, and cell locators.
- Structured predicates.
- Reconstruction scripts.
- Validation and verification scripts.
- Dataset card, release checklist, and license audit.

Public release must not include:

- Raw PDF/HWP/HWPX files.
- Full extracted document text.
- Long context bundle text.
- Raw API row dumps.
- API keys, cookies, or private credentials.
- Screenshots or rendered pages.
- Private finance, insurance, or construction-company materials unless explicit permission is
  obtained and recorded.

Official law/statute text can be used conservatively because National Law Information Center states
that law information falling under Korean Copyright Act Article 7, and works whose economic rights
are fully held by the Ministry of Government Legislation, may be freely used. Public-work reuse
should still record source and conditions. KOGL Type 1 allows commercial and derivative use with
attribution, but each source must be checked by type.

Reference URLs:

- National Law Information Center copyright/use notice: https://law.go.kr/LSW/lawPetitionForm.do?menuId=13&query=&subMenuId=79
- KOGL type guide: https://www.kogl.or.kr/info/faqList.do?cPage=1&dataGroup=2&mstIdx=KOGL_005
- Public Data Portal: https://www.data.go.kr/

## 6. Work Packages

### WP0. Fix v0.4 Public Artifacts

Before expanding, remove release inconsistencies:

- Fix stale `DATASET_CARD.md` language that still says the QA set is small or final bundles are not
  materialized.
- Fix any incorrect Public Data Portal dataset IDs in `docs/public_reconstruction.md`.
- Polish obvious QA wording artifacts such as malformed punctuation and grammar.
- Run the existing validation suite after changes.

### WP1. Source Expansion

Build a new target manifest for v0.5.

Required manifest fields:

- `announcement_id`
- `provider`
- `official_page_url`
- `attachment_url`
- `attachment_filename`
- `document_format`
- `region_sido`
- `region_sigungu`
- `announcement_date`
- `housing_type`
- `program_type`
- `source_license_basis`
- `redistribution_policy`
- `acquisition_status`
- `notes`

Inclusion rules:

- Official provider page or official public portal only.
- Public housing or public-agency housing announcement preferred.
- Raw file must be downloadable without login.
- If source has unclear copyright status, keep raw internal and release URL/locator only.
- Drop private construction-company marketing brochures unless explicit permission exists.

### WP2. Extraction and Table Cell Grids

Generalize extraction beyond LH.

Required output per announcement:

- `document_pages.jsonl`
- `numeric_facts.jsonl`
- `tables.jsonl`
- `table_cells.jsonl`
- `extraction_report.json`

Table cell fields:

- `table_id`
- `page_id`
- `row_index`
- `col_index`
- `row_header`
- `col_header`
- `cell_text`
- `normalized_value`
- `unit`
- `bbox` if available
- `source_format`
- `confidence`

The goal is to support real Korean announcement tables, not only plain-text snippets.

### WP3. Source and License Audit

Every source must have an audit record.

Required checks:

- Official source URL resolves.
- Provider is public or rights are explicitly recorded.
- Raw redistribution decision is recorded separately from benchmark use.
- If KOGL is used, record type.
- If relying on Copyright Act Article 7, state why the source is an official notice/statute/rule.
- Public release fields do not include long excerpts.

### WP4. QA Expansion

v0.5 should increase realism, not only count.

Target family distribution:

| Family | v0.5 Target |
|---|---:|
| table_numeric_reasoning | 400 |
| long_context_retrieval | 300 |
| long_distance_retrieval | 250 |
| cross_document_legal_reasoning | 250 |
| cross_source_aggregation | 250 |
| multi_document_comparison | 200 |
| format_robustness | 150 |
| answerability_detection | 200 |
| correction_notice_reasoning | 150 |
| eligibility_reasoning | 200 |
| schedule_reasoning | 150 |
| total | 2,500 target, 2,000 minimum |

Realistic task examples:

- Compare unit type prices and areas.
- Compute deposit/monthly-rent conversion.
- Find application, document-submission, winner-announcement, and contract dates.
- Determine whether a question is unanswerable from a given bundle.
- Compare original and correction notices.
- Combine announcement criteria with applicable statute/rule.
- Aggregate official table rows by region/month/provider.
- Distinguish similar housing types and applicant categories.

### WP5. Split Design

Use announcement-level split. Never put QA from the same announcement in both dev and test.

Recommended split:

- `dev`: public debugging and prompt development.
- `test_public`: public evaluation with visible answers.
- `test_hidden`: evaluation answers kept internal.
- `ood_provider`: test set from providers absent from dev.
- `ood_region`: test set from regions absent from dev.
- `ood_year`: test set from the newest year when feasible.

Record split assignment in both the QA item and announcement manifest.

### WP6. Context Bundles

Rebuild bundles after source expansion.

Targets:

- 32k, 64k, 128k, 256k, 512k all balanced.
- Evidence positions: early, middle, late, multi.
- Bundle types: announcement-heavy, law-heavy, table-heavy, mixed distractor-heavy.
- Distractors should be licensed public materials from the same domain.

Avoid overfitting:

- No single announcement should dominate non-table QA.
- No single provider should dominate test splits.
- Each context tier should include multiple providers and housing types.

### WP7. Verification

Extend validators and verifiers so expansion cannot silently lower quality.

Required automated checks:

- Source IDs resolve.
- Provider/region/type fields are present.
- Page IDs, row IDs, and cell IDs exist.
- Predicates recompute answers.
- Gold terms appear in cited page/cell/statute evidence.
- Bundle IDs exist.
- Evidence position matches bundle manifest.
- No duplicate or near-duplicate questions.
- No same-announcement split leakage.
- No source/provider dominance beyond threshold.
- No raw text, secret, or long-excerpt leakage.

Required human review:

- 300+ stratified items.
- At least 50 from agent-authored legal/multi-document families.
- All new task families receive review coverage.
- Report pass/fail rate by family and by provider.

### WP8. Documentation and Reproducibility

Update public docs after expansion:

- `README.md`
- `DATASET_CARD.md`
- `docs/public_reconstruction.md`
- `docs/release_checklist.md`
- `docs/source_selection_and_license_audit.md`
- `docs/v0.5_batch_report.md`

The v0.5 batch report must explicitly state:

- what changed from v0.4;
- source/provider/region/type coverage;
- QA counts by family and split;
- context-tier distribution;
- human-review status;
- automated gate results;
- known limitations;
- what is still internal and why.

## 7. Stop Conditions

Stop and report instead of forcing completion if:

1. Fewer than 30 official announcements can be acquired without login.
2. Provider pages cannot be linked to stable official URLs.
3. Raw files or API rows would need to be publicly redistributed to run the benchmark.
4. Source copyright status is unclear and cannot be conservatively handled by URL/locator release.
5. Table extraction fails for more than 50% of selected table-heavy documents.
6. Automatic verification fails above 1%.
7. Human review finds serious grounding errors above 5%.
8. Split leakage cannot be eliminated without discarding a large portion of QA.

## 8. Priority Order

Do the work in this order:

1. Fix v0.4 public artifact inconsistencies.
2. Build v0.5 source target manifest.
3. Acquire and extract provider-diverse documents.
4. Add table cell-grid extraction.
5. Create announcement-level splits.
6. Generate QA with realistic Korean housing tasks.
7. Build balanced context bundles.
8. Verify automatically.
9. Draw and complete human review.
10. Update public docs and release checklist.
