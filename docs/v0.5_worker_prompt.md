the repository root and can run a dynamic workflow.

- Polish obvious QA wording defects if they are generated by scripts; do not hand-edit many QA rows

  announcement_id, provider, official_page_url, attachment_url, attachment_filename, document_format,

- Store raw files only under workspace_local/raw/<provider-or-source>/.

- For each announcement, produce:
  document_pages.jsonl, numeric_facts.jsonl, tables.jsonl, table_cells.jsonl, extraction_report.json.

  and exclude low-confidence cells from gold QA.

- Add split assignment at announcement level.
- Ensure no same announcement appears in multiple evaluation splits.

  dev, test_public, test_hidden, ood_provider, ood_region, ood_year.

- Create data/qa_v0.5_candidates.jsonl.

- Agent-authored QA is allowed only if it passes adversarial review, grounding checks, and is included

- Add realistic Korean housing task families:
  table_numeric_reasoning, long_context_retrieval, long_distance_retrieval,
  cross_document_legal_reasoning, cross_source_aggregation, multi_document_comparison,
  format_robustness, answerability_detection, correction_notice_reasoning, eligibility_reasoning,
  schedule_reasoning.
- Avoid cloze-only overproduction. Use natural practical questions where possible.
- No duplicate or near-duplicate questions.

WP6. Rebuild context bundles

- Build v0.5 bundles under workspace_local/processed/bundles-v05/.
- Use 32k, 64k, 128k, 256k, 512k tiers.
- Balance evidence positions: early, middle, late, multi.

WP7. Extend verification

- Extend validate_dataset.py, verify_qa.py, and check_public_release_readiness.py or add v0.5-specific

  source IDs resolve; provider/region/type fields exist; page IDs/row IDs/cell IDs exist; predicates

  bundle manifest; no duplicate/near-duplicate questions; no same-announcement split leakage; no

- Create docs/v0.5_batch_report.md.

Run at minimum:

python3 scripts/validate_dataset.py
python3 scripts/verify_qa.py
python3 scripts/check_public_release_readiness.py --allow-dev

If you add v0.5-specific scripts, run those too and document exact commands in docs/v0.5_batch_report.md.

Stop conditions:

Stop and report instead of forcing completion if any of these occur:

1. Fewer than 30 official announcements can be acquired without login.
2. Provider pages cannot be linked to stable official URLs.
3. Raw files or API rows would need to be publicly redistributed to run the benchmark.
4. Source copyright status cannot be handled conservatively.
5. Table extraction fails for more than 50% of selected table-heavy documents.
6. Automatic verification fails above 1%.
7. Human review, if performed, finds serious grounding errors above 5%.
8. Split leakage cannot be eliminated without discarding a large portion of QA.

Final response format:

- State whether v0.5 is public-ready, v0.5-dev, or blocked.
- Give exact counts: sources, providers, regions, housing types, QA, splits, bundles, human-review sample.
- List validation commands and pass/fail results.

- List honest limitations and next batch priorities.
